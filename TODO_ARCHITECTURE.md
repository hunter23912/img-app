# 架构优化 TODO

目标：在不扩大项目复杂度的前提下，逐步把当前图片生成应用整理成更容易维护、扩展和发布的结构。

## 1. 当前问题

- 前端主要逻辑集中在 `src/App.tsx`，页面、状态、接口调用、交互和样式耦合较重。
- 后端主要逻辑集中在 `backend/main.go`，配置、路由、请求代理、图片处理和错误响应需要拆分。
- UI 目前依赖较轻，但组件层次不足，后续增加压缩、去水印等工具后容易继续膨胀。
- 图片处理能力还没有形成统一模块，生成、编辑、压缩、去水印后续应共享输入校验、文件限制、错误处理和响应结构。

## 2. 前端重构

- 拆分页面结构：
  - `src/pages/GeneratePage.tsx`
  - `src/pages/EditPage.tsx`
  - `src/pages/ToolsPage.tsx`
- 拆分通用组件：
  - 图片上传组件
  - 参数表单组件
  - 生成结果预览组件
  - 加载、错误、空状态组件
- 抽离 API 调用：
  - `src/api/client.ts`
  - `src/api/images.ts`
- 抽离类型定义：
  - `src/types/image.ts`
  - `src/types/api.ts`
- 抽离状态和工具函数：
  - 表单状态
  - 图片文件校验
  - base64 / Blob / 下载处理
- UI 库优化：
  - 保留 Tailwind 作为基础样式系统。
  - 评估 daisyUI 是否继续承担表单、按钮、弹窗等组件风格。
  - 如需更完整组件能力，再考虑引入 Radix UI / Headless UI，不先引入重型组件库。

## 3. 后端重构

- 拆分目录结构：
  - `backend/internal/config`：环境变量和默认配置
  - `backend/internal/httpapi`：路由、请求解析、响应封装
  - `backend/internal/provider`：大模型图片接口代理
  - `backend/internal/imageops`：图片压缩、去水印等本地图片处理
  - `backend/internal/errors`：统一错误类型和 HTTP 错误响应
- 保留标准库优先，只有在图片处理确实需要时再引入第三方 Go 包。
- 为请求体大小、图片格式、超时时间、并发数量增加统一限制。
- 将 CORS、请求日志、错误响应做成中间件。

## 4. 新功能规划

- 图片压缩：
  - 保持原图尺寸，不做缩小尺寸压缩。
  - 支持 JPG / PNG 输出。
  - JPG 支持质量参数，PNG 优先做重编码和元数据剥离。
  - 前端显示压缩前后大小和下载按钮。
- 去水印：
  - 第一阶段做“涂抹区域 + 后端处理”的基础流程。
  - 前端支持框选或画笔标记区域。
  - 后端先预留 `imageops.RemoveWatermark` 接口，具体实现可后续接入模型或图像修复服务。
- 工具入口：
  - 保持生成图片为主入口。
  - 压缩、去水印放到工具区，避免首页一次性堆太多控件。

## 5. 推荐执行顺序

1. 后端拆分配置、路由、响应结构，保持行为不变。
2. 前端拆分 API、类型、核心组件，保持 UI 基本不变。
3. 调整页面布局和组件样式，统一视觉规范。
4. 加入图片压缩功能。
5. 加入去水印的前端标记流程和后端接口占位。
6. 根据实际去水印方案接入本地算法、第三方服务或大模型接口。

## 6. 验收标准

- `pnpm build` 可通过。
- Go 后端可编译运行。
- 原有图片生成和编辑功能不回退。
- 新增模块边界清晰，单个文件不继续无节制变大。
- 新功能先小步上线，避免一次性引入过多依赖和复杂状态。
