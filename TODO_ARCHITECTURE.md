# 架构优化 TODO

目标：在不扩大项目复杂度的前提下，逐步把当前图片生成应用整理成更容易维护、扩展和发布的结构。

## 1. 当前问题

- 前端已经拆出组件、hooks、API 层，但 `src/App.tsx` 仍承担 tab、生成/编辑表单状态和提交流程，后续可继续拆页面层。
- 后端 handler 已按功能拆分，但还没有形成 `internal/config`、`internal/httpapi`、`internal/imageops` 等正式包结构。
- 图片处理能力已经覆盖生成、编辑、压缩、批量压缩、去水印，但输入校验、文件大小限制、中文错误提示仍比较分散。
- 去水印已有本地算法闭环，但当前是基础邻域扩散修复，适合简单背景，小面积水印；复杂纹理下效果有限。

## 2. 前端重构

- 拆分页面结构：
  - `src/pages/GeneratePage.tsx`
  - `src/pages/EditPage.tsx`
  - `src/pages/ToolsPage.tsx`
- 拆分通用组件：
  - 图片上传组件
  - 参数表单组件
  - 生成结果预览组件
  - 加载、错误、空状态组件
- 抽离 API 调用：
  - `src/api/client.ts`
  - `src/api/images.ts`
- 抽离类型定义：
  - `src/types/image.ts`
  - `src/types/api.ts`
- 抽离状态和工具函数：
  - 表单状态
  - 图片文件校验
  - base64 / Blob / 下载处理
- UI 库优化：
  - 保留 Tailwind 作为基础样式系统。
  - 评估 daisyUI 是否继续承担表单、按钮、弹窗等组件风格。
  - 如需更完整组件能力，再考虑引入 Radix UI / Headless UI，不先引入重型组件库。

## 3. 后端重构

- 拆分目录结构：
  - `backend/internal/config`：环境变量和默认配置
  - `backend/internal/httpapi`：路由、请求解析、响应封装
  - `backend/internal/provider`：大模型图片接口代理
  - `backend/internal/imageops`：图片压缩、去水印等本地图片处理
  - `backend/internal/errors`：统一错误类型和 HTTP 错误响应
- 保留标准库优先，只有在图片处理确实需要时再引入第三方 Go 包。
- 为请求体大小、图片格式、超时时间、并发数量增加统一限制。
- 将 CORS、请求日志、错误响应做成中间件。

## 4. 新功能规划

- 图片压缩：
  - 已支持单图压缩：保持原图尺寸，输出 JPG / PNG。
  - 已支持批量压缩：多图上传，后端压缩后返回 zip，包含 `manifest.json` 明细。
  - 后续可做压缩前后对比视图、批量结果列表、失败文件更友好的前端展示。
- 去水印：
  - 已支持“上传图片 + 画笔标记 mask + 后端本地处理”。
  - 当前后端使用本地遮罩邻域扩散修复，输出 PNG。
  - 后续可加入边缘扩展、修复强度、撤销/重做、前后对比视图。
- 工具入口：
  - 保持生成图片为主入口。
  - 压缩、去水印放到工具区，避免首页一次性堆太多控件。

## 5. 推荐执行顺序

1. 增加压缩和去水印的前后对比视图，优先复用为通用 `ImageCompare` 组件。
2. 完善批量压缩结果展示：显示成功/失败文件列表，支持单独下载失败原因或展开 `manifest.json` 摘要。
3. 做统一图片校验和中文错误提示：格式、大小、尺寸、空 mask、全部压缩失败等场景。
4. 继续前端页面拆分：`ImagePage`、`CompressPage`、`WatermarkPage`，降低 `App.tsx` 复杂度。
5. 后端整理包结构：先移动本地图片处理到 `internal/imageops`，再拆 config、响应和路由。
6. 部署方向按实际场景选择：服务器部署优先 Nginx 托管 `dist` + 反代 `/api`；本地分享再考虑 Go embed 前端资源。

## 6. 验收标准

- `pnpm build` 可通过。
- Go 后端可编译运行。
- 原有图片生成和编辑功能不回退。
- 新增模块边界清晰，单个文件不继续无节制变大。
- 新功能先小步上线，避免一次性引入过多依赖和复杂状态。
